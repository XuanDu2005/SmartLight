## 1. Purpose

This document defines SmartLight's **CI/CD architecture**: pipeline, branching strategy, release process, and deployment automation. Built on **GitHub Actions**.

---

## 2. Pipeline Principles

1. **Fast feedback** — PR builds < 5 min.
2. **Trustworthy CI** — fail on any gate.
3. **Reproducible builds** — same commit = same artifact.
4. **No manual deploys** — automated via GH Actions.
5. **Trunk-based** with short-lived feature branches.

---

## 3. Branch Strategy

### 3.1 Branch Types

| Branch | Purpose | Lifetime | Protection |
|---|---|---|---|
| `main` | Production-ready | Permanent | Strict |
| `develop` (V1.1+) | Integration branch | Permanent | Strict |
| `feature/*` | New work | Short (<3 days) | PR required |
| `fix/*` | Bug fix | Short | PR required |
| `release/*` | Release prep | Short | Strict |
| `hotfix/*` | Emergency | As needed | Fast-tracked |
| `chore/*` | Tooling, deps | Short | Normal |

### 3.2 V1 Branching

V1 uses **trunk-based development**:
- `main` is always deployable
- Short-lived `feature/*` branches
- Squash-merge to `main`
- No `develop` branch

### 3.3 V1.5+ Branching (Optional GitFlow)

When release cadence slows:
- `develop` for integration
- `release/x.y` branches for stabilization
- `main` for production

> Recommendation: stay trunk-based; tag releases on `main`.

### 3.4 Branch Protection

`main`:
- ✓ Require PR reviews (1+)
- ✓ Require status checks (CI green)
- ✓ Require up-to-date branch
- ✓ No force-push
- ✓ No deletion
- ✓ Linear history required

## 4. Commit Convention

Adopt **Conventional Commits** for semantic versioning:

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

| Type | Use |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation |
| `style` | Formatting (no logic change) |
| `refactor` | Code change (no behavior change) |
| `perf` | Performance |
| `test` | Adding/updating tests |
| `chore` | Tooling, deps |
| `build` | Build system |
| `ci` | CI configuration |
| `revert` | Reverting changes |

Examples:
```
feat(order): add return request endpoint
fix(payment): handle VNPay timeout correctly
docs(api): document webhook signature verification
```

### 4.1 Breaking Changes

`feat!: change response shape` or footer `BREAKING CHANGE: ...`

## 5. Release Strategy

### 5.1 Versioning

- **Semantic Versioning 2.0** (`MAJOR.MINOR.PATCH`)
- Starting at `1.0.0` for MVP
- Pre-release tags: `1.1.0-rc.1`

### 5.2 Release Cadence

| Release Type | Cadence |
|---|---|
| Major | When breaking changes needed (V2+) |
| Minor | Monthly (feature releases) |
| Patch | As needed (hotfixes) |

### 5.3 Release Process

```
1. Trunk commit on main
2. Tag v1.4.2
3. CI builds and deploys to production
4. Smoke tests run
5. Release notes auto-generated from commits
6. Slack notification
```

### 5.4 Release Notes

Generated by `release-please` or `semantic-release` from conventional commit messages.

## 6. GitHub Actions Workflows

### 6.1 PR Workflow (`pr.yml`)

Triggered on PR to `main`:

```
1. Setup
   - Checkout
   - Setup Node.js 20
   - Install pnpm + deps
   - Cache deps

2. Quality Gates (parallel)
   - Lint (ESLint, Prettier check)
   - Typecheck (tsc --noEmit)
   - Unit tests (Jest, with coverage)
   - OpenAPI diff (against main)
   - Architecture test (madge --circular)
   - Secret scan (gitleaks)
   - CodeQL (V1.5+)
   - Dependency audit (npm audit)

3. Build
   - Build api
   - Build storefront
   - Build admin

4. Comment PR with summary
   - Coverage delta
   - Bundle size delta
   - OpenAPI diff
```

### 6.2 Deploy Workflow (`deploy.yml`)

Triggered on push to `main`:

```
1. Same as PR gates (re-run)
2. Build images
   - smartlight/api:{version}-{sha}
   - smartlight/api:{version} (tag)
   - smartlight/api:latest (latest stable)
3. Push to GitHub Container Registry
4. Apply DB migration to staging
5. Deploy to staging
6. Run e2e tests against staging
7. Apply DB migration to production (manual approval)
8. Deploy to production
9. Run smoke tests against production
10. Notify Slack
```

### 6.3 Hotfix Workflow (`hotfix.yml`)

Manual trigger via `workflow_dispatch`:

```
- Build, test, deploy quickly
- Bypasses full e2e for speed
- Manual approval required for production
```

### 6.4 Scheduled Jobs (`.github/workflows/scheduled.yml`)

```
- Daily: dependency audit
- Weekly: dependency updates (Dependabot)
- Weekly: backup verification
```

## 7. Environments in GitHub

| Environment | URL | Secrets |
|---|---|---|
| `preview` | per-PR URL | All GH secrets |
| `staging` | staging.smartlight.vn | Staging-only secrets |
| `production` | smartlight.vn | Production-only secrets |

### 7.1 Environment Rules

- `production` requires manual approval (1+ approver)
- Secret access scoped per environment
- Deployment branches restricted (only `main`)

## 10. Quality Gates Definition

### 10.1 Coverage

| Module | Required Coverage |
|---|---|
| Domain | 90% line |
| Application | 80% line |
| Infrastructure | 60% line (interfaces) |
| Interface | 70% line |
| Overall | 80% |

Coverage decrease > 1% fails the build.

### 10.2 Bundle Size

- Storefront: < 500 KB initial JS (gzipped)
- Admin: < 800 KB initial JS
- API Docker image: < 500 MB

> Size increase > 10% warns; > 25% fails.

### 10.3 Lint & Type

- ESLint: 0 errors
- Prettier: 0 issues
- TypeScript: 0 errors with `strict: true`

## 11. Test Automation

### 11.1 Test Pyramid

```
       / E2E \           ← Playwright (10s of scenarios)
      /───────\
     /  Int.   \         ← supertest (100s of cases)
    /───────────\
   /  Unit Tests \       ← Jest (1000s of tests)
  /───────────────\
```

### 11.2 V1 Test Coverage Targets

| Layer | Test Type | Coverage Goal |
|---|---|---|
| Domain | Unit | 90%+ |
| Application | Unit (with ports mocked) | 80%+ |
| Infrastructure | Integration (Prisma test DB) | 60%+ |
| Interface | E2E (supertest) | 70%+ |

### 11.3 E2E Coverage

- Customer: browse → cart → checkout → payment confirmation
- Admin: login → CRUD product → manage order
- Webhook: VNPay happy path; signature failure

### 11.4 Test Data

- Fixtures per test (factories)
- No shared mutable state
- DB rollback per test

## 12. Dependency Management

## 16. Rollback Procedure

### 16.1 Application Rollback

Via Railway / Vercel dashboard or CLI:

```
railway up --rollback
vercel rollback
```

### 16.2 Database Rollback

Forward-only migrations. If needed:

```
- Write a new migration to revert
- Apply via CI
- Validate
```

> Never manually edit DB.
