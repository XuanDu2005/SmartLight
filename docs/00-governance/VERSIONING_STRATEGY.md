## 1. Purpose

This document defines the **versioning strategy** for SmartLight. It applies to:

- The **product platform** as a whole (release versions).
- The **monorepo packages** (internal package versions).
- The **database schema** (migration strategy).
- The **API contract** (public and internal versions).

Goals:

1. Make release scope and impact **immediately understandable**.
2. Enable **safe, predictable upgrades** for users and developers.
3. Allow **independent evolution** of internal packages without breaking consumers.
4. Support **deprecation** as a first-class lifecycle stage.

## 2. Product Release Versioning (Semantic Versioning)

SmartLight product releases follow **Semantic Versioning 2.0.0**:

```
MAJOR.MINOR.PATCH
  │     │     │
  │     │     └── backward-compatible bug fixes
  │     └──────── backward-compatible new features
  └────────────── incompatible API or behavior changes
```

### 2.1 Version Increment Rules

| Change | Bump |
| --- | --- |
| Breaking change to public API or contract | **MAJOR** |
| Breaking change to data model affecting existing records | **MAJOR** |
| Removed deprecated feature | **MAJOR** |
| New backward-compatible feature | **MINOR** |
| New backward-compatible API endpoint | **MINOR** |
| New optional input field with safe default | **MINOR** |
| Bug fix with no behavior change for callers | **PATCH** |
| Documentation-only changes | No version bump |
| Internal-only refactor | No version bump |
| Performance improvement with no behavior change | **PATCH** |

### 2.2 Initial Versions

| Version | Significance |
| --- | --- |
| `0.1.0` | Internal alpha — engineering only |
| `0.5.0` | Closed beta — invited users |
| `1.0.0` | Public launch — first GA single-vendor release |
| `1.x.y` | Single-vendor enhancements |
| `2.0.0` | Marketplace launch (V2) — multi-seller |

### 2.3 Pre-Release Identifiers

Allowed for pre-release versions:

- `1.0.0-alpha.1`
- `1.0.0-beta.2`
- `1.0.0-rc.1`

Pre-release versions are **not deployed to production** unless explicitly approved.

## 3. API Versioning

### 3.1 URL Versioning

External public APIs are versioned in the URL path:

```
/api/v1/products
/api/v1/orders
```

- `/api/v1/` is the **current** stable API.
- Future breaking changes introduce `/api/v2/` while `/api/v1/` is maintained.
- Deprecated endpoints return `Sunset` and `Deprecation` HTTP headers.

### 3.2 Internal Module APIs

- Internal NestJS module-to-module communication is **not versioned** in URL paths.
- It uses typed contracts from `@smartlight/contracts`.
- Breaking changes are coordinated through the package versioning (see Section 4).

### 3.3 Deprecation Headers

When an endpoint is deprecated:

| Header | Value |
| --- | --- |
| `Deprecation` | RFC-8594 `true` or date |
| `Sunset` | RFC-8594 date |
| `Link` | `<https://api.smartlight.vn/docs/v2/...>; rel="successor-version"` |

Deprecation period: **minimum 6 months** before removal.

## 4. Monorepo Package Versioning (Changesets)

Internal packages (`@smartlight/*`) use **[Changesets](https://github.com/changesets/changesets)** for versioning.

### 4.1 Why Changesets

- Decouples package versioning from product release timing.
- Generates accurate changelogs automatically.
- Coordinates cross-package changes.

### 4.2 Changeset Format

Every PR that affects a published package includes a `.changeset/<random-name>.md` file:

```md
---
'@smartlight/contracts': minor
'@smartlight/ui': patch
---

Add `OrderDto.shippingCarrier` and update UI badge styling.
```

### 4.3 Semver Mapping in Changesets

| Bump | When |
| --- | --- |
| `major` | Breaking change to a package's public API |
| `minor` | Backward-compatible feature added |
| `patch` | Backward-compatible fix |

### 4.4 Release Process

1. PRs add changesets describing their package impact.
2. The "Version Packages" PR is auto-generated by Changesets bot.
3. After approval and merge, new package versions are published.
4. Changelog updated.

### 4.5 Internal-Only Packages

Packages marked `"private": true` in `package.json` are **not published** and do not require changesets.

## 5. Database Schema Versioning

### 5.1 Migration as Version

- Each Prisma migration represents a **schema version**.
- Migrations are append-only and never edited after merge to `main`.
- Migration filenames are timestamps + descriptive name: `20260702_add_order_status_enum`.

### 5.2 Schema Compatibility Rules

| Migration Type | Safe? | Action |
| --- | --- | --- |
| Add column with safe default | Yes | Single deploy |
| Add column with no default | Yes (nullable) | Single deploy |
| Add table | Yes | Single deploy |
| Add index concurrently | Yes | Single deploy |
| Drop column | No (destructive) | Two-phase deploy + ADR |
| Rename column | No (destructive) | Two-phase deploy + ADR |
| Change column type | No (destructive) | Two-phase deploy + ADR |
| Drop table | No (destructive) | Two-phase deploy + ADR |

### 5.3 Two-Phase Destructive Migrations

1. **Phase 1:** Add new structure, dual-write data, keep old structure.
2. **Phase 2 (next release):** Migrate readers to new structure, then drop old structure.

### 5.4 Migration Versioning

- Database "version" = the latest applied migration timestamp.
- Documented in `apps/api/prisma/MIGRATIONS.md`.

## 6. Frontend Build Versioning

### 6.1 Build Metadata

Every frontend build embeds:

```ts
export const BUILD_INFO = {
  version: '1.4.0',
  commit: 'a1b2c3d',
  builtAt: '2026-07-02T12:34:56Z',
  environment: 'production',
};
```

- Surfaced in `package.json`, in-app About modal, and `/_/version.json` endpoint.
- Enables instant identification of running version.

### 6.2 Asset Versioning

- Vercel handles immutable asset URLs with content hashes.
- Long-term caching safe.

## 7. Backend Service Versioning

### 7.1 Health and Version Endpoint

```
GET /_/health     → 200 OK with service health
GET /_/version    → { name, version, commit, builtAt, env }
```

### 7.2 Container Image Tags

- Image tags follow: `vX.Y.Z`, `vX.Y`, `latest`, `<git-sha>`.
- Production deploys use immutable `vX.Y.Z` tags.
- `latest` is only updated by CI on `main`.

## 8. Release Cadence

| Release Type | Frequency | Examples |
| --- | --- | --- |
| **Major** | Yearly or on business-model change | `1.0.0`, `2.0.0` |
| **Minor** | Monthly | `1.1.0`, `1.2.0` |
| **Patch** | As needed | `1.1.1`, `1.1.2` |
| **Hotfix** | Within 24h of incident | `1.1.2-hotfix.1` |

### 8.1 Release Freeze

- Code freeze for `main` during **3 business days** before a major release.
- Only Sev-1 fixes allowed during freeze.
- Freeze declared by Tech Lead and lifted upon release.

## 9. Changelog Discipline

### 9.1 User-Facing Changelog

- Maintained in `docs/90-changelog/CHANGELOG.md`.
- Auto-generated from changesets where possible.
- Curated by Product for the public release notes section.

### 9.2 Internal Changelog

- Generated automatically by Changesets per package.
- Detailed in `packages/<pkg>/CHANGELOG.md`.

### 9.3 Changelog Categories

| Category | Description |
| --- | --- |
| **Features** | New user-facing capabilities |
| **Improvements** | Enhancements to existing features |
| **Fixes** | Bug fixes |
| **Performance** | Performance improvements |
| **Security** | Security patches |
| **Breaking Changes** | With migration notes |
| **Deprecations** | With timeline |
| **Internal** | Engineering-only changes |

## 10. Deprecation Policy

### 10.1 API Deprecation

1. Mark endpoint as `@Deprecated()` in code.
2. Emit `Deprecation` and `Sunset` headers.
3. Update OpenAPI spec with `deprecated: true` and `x-sunset`.
4. Add entry to changelog.
5. Notify integrators (email, in-app banner).
6. Remove only after sunset date.

### 10.2 Feature Deprecation

1. Announce in product changelog.
2. Show in-app banner.
3. Maintain feature for at least one minor release.
4. Remove in next major release.

### 10.3 Minimum Deprecation Window

| Surface | Window |
| --- | --- |
| Public API endpoint | 6 months |
| Feature flag | Until feature is GA + 1 minor release |
| Internal module contract | Next minor release |
| Admin capability | 1 minor release with notice |

## 11. Backward Compatibility Commitments

### 11.1 API Backward Compatibility

Within a major version:

- New endpoints may be added.
- New optional fields may be added to responses.
- New optional query parameters may be added.
- Endpoint behavior for documented inputs must remain consistent.
- Default values for new optional fields must be safe.

Forbidden without MAJOR bump:

- Removing endpoints.
- Renaming fields.
- Changing field types.
- Tightening validation (e.g., rejecting previously accepted values).
- Changing error codes for existing scenarios.

### 11.2 Database Backward Compatibility

Within a major version:

- Existing tables and columns preserved (unless explicitly deprecated and announced).
- Application tolerates both old and new column shapes during transitions.

## 12. Version Bump Decision Matrix

| Change | Bump |
| --- | --- |
| Add new public endpoint | MINOR |
| Add new optional request field with safe default | MINOR |
| Add new response field | MINOR |
| Add new database column with safe default | MINOR (with migration) |
| Tighten auth on existing endpoint | MAJOR (security carve-out: PATCH + ADR if urgently required) |
| Change endpoint URL | MAJOR |
| Rename response field | MAJOR |
| Change HTTP status code of existing scenario | MAJOR |
| Drop column | MAJOR |
| Change error code | MAJOR |
| Fix bug in business logic | PATCH |
| Refactor without behavior change | No bump |
| Update dependency to compatible version | No bump |

### 12.1 Security Carve-Out

A security fix may be released as a PATCH even if it changes an error code, provided:

- An ADR documents the change.
- The security gain outweighs the compatibility cost.
- Notification is sent to integrators.

## 13. Tag and Release Process

### 13.1 Release Steps

1. Engineering Lead confirms all merged PRs for the release are in `main`.
2. CI builds production artifacts.
3. Smoke tests run on staging.
4. QA signs off.
5. Tech Lead creates the Git tag (`vX.Y.Z`).
6. Production deploy triggered.
7. Release notes published.
8. Tag broadcast to team channels.

### 13.2 Tag Immutability

Tags are immutable. Corrections require a new tag/version.

## 14. Versioning and AI Agents

- AI agents must read this document before contributing to versioning-sensitive changes.
- AI agents must not bump versions directly — Changesets bot or humans handle it.
- AI agents must include the SemVer impact in their PR description when changes affect public APIs.
- AI agents must not modify `CHANGELOG.md` directly; Changesets handles it.
